The Tempera SDK exposes **one uniform surface in three languages** — TypeScript
(`@tempera/sdk`), Python (`tempera-sdk`), and Rust (`tempera-sdk`). The primary
workflow connects the control plane, Tempo browser sessions, Human Data review,
and Palette measurement through an onboarding-provisioned integration and
correlation path. A single manifest, `surface.json`, also preserves the
full versioned private contract inventory so the packages and reference docs cannot
drift apart.

> **Private access.** Hosted Tempera services are available through design-partner onboarding.
> Use the issuer, credentials, environment, and product URLs provisioned for your workspace;
> start in staging unless Tempera explicitly approves another target.

## The browser-agent quality loop

1. **Control plane** — authenticate and select the provisioned workspace.
2. **Tempo** — run and record the browser session.
3. **Human Data** — review the provisioned session and trace evidence, record a
   decision, and return a candidate case to the quality loop.
4. **Palette** — hold and measure the corresponding trace.

Tempo session creation and Palette evidence are separate contracts. A Tempo
session does not by itself prove a Palette trace exists; onboarding supplies the
integration and correlation path for corresponding Palette evidence.

Human Data is currently a provisioned review workflow, not a published typed SDK
operation. The SDK does not claim a public Human Data HTTP path.

## Quickstart

After private onboarding, authenticate with the issuer and API key supplied for
your workspace, then connect the typed parts of the loop. These examples start
with the staging preset. The examples call each typed surface independently; your
onboarding-provisioned integration correlates the Tempo session with any
corresponding Palette evidence for Human Data review.

<CodeGroup>

```typescript TypeScript
// Package access and environment values are supplied during onboarding.
import { TemperaAuth, createTemperaClient } from "@tempera/sdk";

const issuerUrl = process.env.TEMPERA_ISSUER_URL;
const apiKey = process.env.TEMPERA_API_KEY;
const tenantId = process.env.TEMPERA_TENANT_ID;
if (!issuerUrl || !apiKey || !tenantId) throw new Error("Missing provisioned Tempera settings");

const auth = new TemperaAuth({
  issuerUrl,
  apiKey, // tp_... works at every product
});
const client = createTemperaClient({ auth, environment: "staging" });

const issuerMetadata = await client.controlPlane.discovery();
const session = await client.tempo.createSession({ url: "https://example.com" });
// Session creation does not prove a Palette trace exists. Onboarding supplies
// the integration and correlation path for the corresponding Palette evidence.
// Human Data reviews the provisioned session and trace evidence.
const availableTraces = await client.palette.listTraces({
  tenant_id: tenantId,
  limit: 20,
});
```

```python Python
# Package access and environment values are supplied during onboarding.
import os
from tempera_sdk import TemperaAuth, TemperaClient

auth = TemperaAuth(
    issuer_url=os.environ["TEMPERA_ISSUER_URL"],
    api_key=os.environ["TEMPERA_API_KEY"],  # tp_... works at every product

client = TemperaClient(auth=auth, environment="staging")

issuer_metadata = client.control_plane.discovery()

# Session creation does not prove a Palette trace exists. Onboarding supplies
# the integration and correlation path for the corresponding Palette evidence.

```

```rust Rust
// Crate access and environment values are supplied during onboarding.
use tempera_sdk::{TemperaAuth, TemperaClient};

// The Rust crate is HTTP-less: it builds fully-described RequestSpecs
// (method, URL, headers, JSON body) for your own HTTP client to send.
let issuer_url = std::env::var("TEMPERA_ISSUER_URL")?;
let api_key = std::env::var("TEMPERA_API_KEY")?;
let tenant_id = std::env::var("TEMPERA_TENANT_ID")?;
let auth = TemperaAuth::new(issuer_url).with_api_key(api_key);
let client = TemperaClient::new().with_auth(auth);

let issuer_metadata = client.build_request("control_plane", "discovery", &[])?;
let session = client.build_request(
    "tempo",
    "create_session",
    &[("url", "https://example.com".into())],
)?;
// Session creation does not prove a Palette trace exists. Onboarding supplies
// the integration and correlation path for the corresponding Palette evidence.
// Human Data reviews the provisioned session and trace evidence.
let available_traces = client.build_request(
    "palette",
    "list_traces",
    &[("tenant_id", tenant_id.into()), ("limit", 20i64.into())],
)?;
// Send each RequestSpec with your own HTTP client.
```

Method naming is mechanical across the languages: the manifest's camelCase
operation id in TypeScript (`listTraces`), snake_case in Python (`list_traces`),
and `build_request(product, "list_traces", params)` in Rust. Parameters use wire
names (snake_case) in every language.

## Primary workflow surfaces

These four surfaces define the primary public story. Hosted access still depends
on what Tempera provisions for your workspace.

## Versioned private contract inventory

The SDK also retains typed and passthrough clients for the complete versioned
registry. Entries outside the four-product workflow are **private contract
inventory**: their presence preserves compatibility and reference coverage; it
does not advertise public availability, a live hosted service, or an undocumented
HTTP endpoint. Use one only when Tempera provisions it and supplies its contract.

The full inventory remains available in the API Reference navigation and the
[registry-only contract inventory](/products/other-products).
