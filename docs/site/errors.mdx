Resource APIs converge on the AIP-193 error envelope while protocol and
legacy surfaces retain explicit compatibility. The SDK normalizes them
into one **`TemperaApiError`** with identical fields in every
language (Python uses `request_id`; the HTTP-less Rust crate normalizes
`status`/`code`/`message`/`request_id` from response bodies via
`normalize_error_body` and carries no request context):

| Field | Meaning |
|---|---|
| `status` | HTTP status code of the failed response. |
| `code` | Machine-readable error code, when the wire shape carried one (else null/None). |
| `message` | Human-readable error message; never empty. |
| `requestId` | Server request id, from the body or the `x-request-id` response header. |
| `product` | Product key of the client that made the call. |
| `operation` | Operation id of the typed call (null/None for passthrough requests). |
| `body` | The raw parsed response body, untouched. |

## Normalization rules

Applied in order to the parsed error body:

1. If body.error is an object: code = body.error.status when it is a string, otherwise body.error.code when it is a string; message = body.error.message; requestId = body.error.requestId or body.error.request_id.
2. Else if body.error is a string and body.message is present: code = body.error and message = body.message (legacy or protocol compatibility).
3. Else if body.error is a string: code = null and message = body.error (legacy compatibility).
4. Else: code = null, message = the HTTP status text.
5. requestId also falls back to the x-request-id response header when the body carries none.

## TemperaMcpError

JSON-RPC errors from the [MCP gateway](/mcp-gateway) raise `TemperaMcpError`
(TypeScript and Python) with the gateway's integer `code`, its `message`, and
any `data` member. The Rust crate returns the same information as `McpError`
(`code`, `message`) from `parse_mcp_error(body)`. A non-conformant error
without an integer code is reported uniformly as code `0`.

## MCP gateway error codes

| Name | Code | Meaning |
|---|---|---|
| `planLimit` | `-32002` | Plan limit exceeded (`plan_limit_exceeded`): the workspace's metered `mcp_invocations` allowance is exhausted; upgrade the plan or wait for the meter to reset. |
| `invalidRequest` | `-32600` | The payload is not a valid JSON-RPC 2.0 request. |
| `methodNotFound` | `-32601` | Unknown JSON-RPC method (or unknown tool name on `tools/call`). |
| `invalidParams` | `-32602` | The method was called with invalid or missing parameters. |
| `internalError` | `-32603` | The gateway or an upstream product MCP server failed. |
