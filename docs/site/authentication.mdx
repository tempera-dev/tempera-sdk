## One account, every product

The provisioned Tempera control plane (auth-hub) is the OAuth 2.1 issuer for the whole
fleet. One Tempera account mints one access token per **product audience**,
and central API keys (`tp_...`) work as bearers at every product via central
introspection. The SDK's `TemperaAuth` holds both: per-audience OAuth token
sets and an optional unified API key.

## OAuth2: authorization code + PKCE (S256)

Public clients use the authorization-code flow with PKCE
(`S256` only). The audience is selected with the RFC 8707
`resource` parameter on both the authorize and token requests:

1. **Authorize** — send the user to `/oauth/authorize` with
   `response_type=code`, your `client_id`, a `code_challenge` (S256), and
   `resource=<audience>` (plus optional `scope` and `state`).
2. **Exchange** — POST the returned code to `/oauth/token` with the
   PKCE `code_verifier` and the same `resource`; you receive the audience's
   access + refresh token set.
3. **Refresh** — POST `grant_type=refresh_token` to `/oauth/token`.
   Refresh tokens **rotate**: the newly issued refresh token replaces the old
   one in the SDK's store.
4. **Revoke** — POST the token to `/oauth/revoke` to invalidate it;
   the SDK drops the stored token set.

import { TemperaAuth, createPkcePair } from "@tempera/sdk";

const issuerUrl = process.env.TEMPERA_ISSUER_URL;
const clientId = process.env.TEMPERA_CLIENT_ID;
if (!issuerUrl || !clientId) throw new Error("Missing provisioned Tempera auth settings");

const auth = new TemperaAuth({
  issuerUrl,
  clientId,
});

// 1. Authorize: PKCE S256 + RFC 8707 resource audience selector.
const { verifier, challenge } = await createPkcePair();
const authorizeUrl = auth.buildAuthorizeUrl({
  redirectUri: "https://my-app.example/callback",
  codeChallenge: challenge,
  audience: "tempo",
  scope: ["trace:read", "trace:write"],
});

// 2. Exchange the returned code for the audience's token set.
await auth.exchangeCode({
  code,
  codeVerifier: verifier,
  redirectUri: "https://my-app.example/callback",
  audience: "tempo",
});

// 3. Refresh with rotation: the new refresh token replaces the old one.
await auth.refresh("tempo");

// 4. Revoke and drop the stored token set.
await auth.revoke("tempo");

use tempera_sdk::{AuthorizeUrlParams, TemperaAuth, pkce_pair_from_entropy};

// The crate is HTTP-less: it builds the URLs and form bodies; you POST them
// (content-type: application/x-www-form-urlencoded) with your HTTP client.
let issuer_url = std::env::var("TEMPERA_ISSUER_URL")?;
let client_id = std::env::var("TEMPERA_CLIENT_ID")?;
let mut auth = TemperaAuth::new(issuer_url).with_client_id(client_id.clone());

// 1. Authorize: PKCE S256 + RFC 8707 resource audience selector.
let pkce = pkce_pair_from_entropy(&entropy); // >= 32 cryptographically random bytes
let authorize_url = auth.authorize_url(&AuthorizeUrlParams {
    client_id: &client_id,
    redirect_uri: "https://my-app.example/callback",
    code_challenge: &pkce.challenge,
    audience: "tempo",
    scope: Some("trace:read trace:write"),
    state: None,
});

// 2. Exchange: POST this body at auth.token_url(), then store the response.
let body = auth.code_exchange_body(
    &code,
    &pkce.verifier,
    "https://my-app.example/callback",
    "tempo",
);
// Refresh-token rotation is applied when you store the parsed response:
auth.apply_token_response("tempo", access_token, Some(refresh_token));

// 3. Refresh: POST this body at auth.token_url().
let refresh_form = auth.refresh_body("tempo");

// 4. Revoke: POST this body at auth.revoke_url(); the stored set is dropped.
let revoke_form = auth.revoke_body("tempo");

## tp_ API keys: universal bearers

Workspace API keys minted by the control plane (`createApiKey`, secrets
prefixed `tp_`) are **universal bearers**: every product accepts them via
central introspection, so one key covers every audience. Give `TemperaAuth`
an `apiKey` / `api_key` / `with_api_key(...)` and skip the OAuth flow
entirely for headless use.

## Bearer resolution order

For each call, the client resolves the bearer in this order:

1. An explicit per-call `bearer` option, when you pass one.
2. Per the operation's auth kind:
   - `none` — no `authorization` header is sent.
   - `account` — the account-session token returned by `createHostedSession` (or
     passed as `accountToken` / `account_token` / `with_account_token`).
   - `oauthResource` — an OAuth access token or central `tp_` API key for the operation's explicit resource audience.
   - `introspectionSecret` — the configured introspection secret
     (`introspectToken` only; server-side).
   - `product` — through `TemperaAuth.bearerFor(audience)`: the audience's
     **access token** if one is stored, else the unified **`tp_` API key**,
     else a configuration error before any request is sent.
